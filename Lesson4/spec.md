# 图书投票网页应用 —— 课程练习设计文档（spec.md）

> 本文档面向**零基础学员**，作为第 4 课 AI Coding 实战练习的设计说明。
> 上一节课，学员已经用 AI Coding 写了一个**纯 HTML 客户端小游戏**（只有前端、刷新即清空、别人看不到你的数据）。
> 这一节课，我们要做一个"**多人一起投票选书**"的网页应用，借此引出**服务器**和**客户端**的概念。

---

## 1. 教学目标

学完本节课后，学员应该能够：

1. 用自己的话说清楚**客户端**和**服务器**分别是什么、各自负责什么。
2. 理解**为什么有些功能只靠浏览器（纯 HTML）做不到**，必须要有服务器。
3. 认识一次**前后端通信**的完整过程：浏览器发请求 → 服务器处理 → 返回数据 → 浏览器展示。
4. 体会**数据持久化**的意义：关掉浏览器再打开，数据还在；别人也能看到我投的票。
5. 能在 AI Coding 助手的帮助下，完成一个最简单的"前端 + 后端 + 数据存储"的小项目。

---

## 2. 概念讲解思路（如何从上节课自然过渡）

### 2.1 用上节课的小游戏作对比

| 对比项 | 上节课：纯 HTML 小游戏 | 本节课：图书投票应用 |
|---|---|---|
| 代码运行在哪里 | 只在你自己的浏览器里 | 浏览器 + 一台共享的服务器 |
| 数据存在哪里 | 内存里，刷新就没了 | 服务器上，所有人共享 |
| 别人能看到你的操作吗 | 看不到 | 能看到（票数会一起变化） |
| 关掉网页再打开 | 一切归零 | 数据还在 |

> **课堂提问引入**：
> "如果我们把上节课的小游戏发给同学，他玩出 100 分，**老师能看到这个分数吗？** 为什么不能？要让老师能看到，需要多一个什么东西？"

### 2.2 用"点奶茶"类比客户端和服务器

- **客户端（浏览器）= 顾客**：负责"看菜单、下单、看结果"，也就是显示界面、收集你点的东西。
- **服务器 = 奶茶店后厨**：负责"记账、做奶茶、把库存写在小本子上"，也就是处理逻辑、保存数据。
- **网络请求 = 点单小票**：顾客把要求写在小票上递进去，后厨做完再把奶茶递出来。

### 2.3 引出本项目

> "今天我们要做的投票选书，就是一个**多人共享的小本子**：
> 每个人在自己的浏览器里点'投票'，但票数要记在**同一本账上**，
> 这本账放在服务器上，所有人看到的都是同一份。"

## 3. 功能需求

### 3.1 必做功能（MVP）

1. **展示书单**：页面打开后，能看到约 5 本书（书名、作者、封面或简介、当前票数）。
2. **投票**：每本书下面有一个"投一票"按钮，点击后票数 +1。
3. **实时看到结果**：投票后，页面上该书的票数立刻更新。
4. **数据共享**：A 同学投的票，B 同学刷新页面后能看到。
5. **数据不丢**：服务器重启后（或下次打开），票数仍然存在。

### 3.2 简化处理（为了让初学者跑通）

- **防重复投票**：本课只做**最简单的本地限制**——用浏览器的 `localStorage` 记住"我投过了"，按钮变灰。
  > （明确告诉学员：这只是"君子协定"，清缓存就能再投。**真正的防作弊需要登录系统**，超出本课范围，作为拓展话题）
- **不做登录注册**：所有人都是匿名用户。
- **不做管理后台**：书单写死在代码里，不做后台增删。

## 4. 技术选型

> **选型原则**：让零基础学员能在一节课内、用 AI Coding 把整个流程跑通，**优先简单、少配置、单文件能启动**。
> **重要约束**：**本项目全程不使用任何数据库**（不用 MySQL / SQLite / MongoDB 等），数据一律以**本地文件**形式保存，让"数据持久化"对初学者可见、可读、可手动修改。

| 层 | 选型 | 理由 |
|---|---|---|
| **前端** | 原生 HTML + CSS + JavaScript（不用框架） | 延续上节课，**不引入 React/Vue 的额外学习成本**；用 `fetch()` 调后端接口足够。 |
| **后端** | **Node.js + Express** | 前后端**同一种语言（JavaScript）**，学员只学一门语法就能打通；Express 启动只需十几行代码，AI Coding 对它非常熟练。 |
| **数据存储** | 一个本地 **JSON 文件**（如 `books.json`） | 不引入数据库；Node.js 原生就支持 JSON（`JSON.parse` / `JSON.stringify`），读写文件用内置的 `fs` 模块即可。学员能直接打开文件看到数据变化，**"持久化"这件事看得见摸得着**。 |
| **运行方式** | `node server.js`，浏览器访问 `http://127.0.0.1:3000` | 一条命令启动，无需部署。 |

> **课堂讲解话术**：
> "今天我们前端和后端**用的是同一种语言**——你已经在浏览器里写过 JavaScript 了，服务器上也是写 JavaScript，只是换了个运行环境（Node.js）。"
> "我们今天不用数据库，就用一个**文本文件当账本**。等你们以后学了数据库，会发现思路是一样的，只是账本更专业、更快而已。"

### 环境准备清单（讲师课前发给学员）

- 安装 **Node.js LTS 版本**（建议 18.x 或 20.x 及以上），自带 `npm`
- 一个文本编辑器（推荐 VS Code）
- 项目初始化命令（课上演示）：
  ```bash
  npm init -y
  npm install express
  ```

### 备选方案（讲师可按学员情况切换）

- 如果学员是 Python 背景：后端可改用 **Python + Flask**，思路完全一致，接口和数据文件不变。
- 如果完全不想装环境：可用在线 IDE（如 Replit / StackBlitz）直接跑 Node.js，但本 spec 默认本地 Node.js。

## 5. 页面与交互设计

### 5.1 页面结构（一个 `public/index.html` 即可）

```
┌─────────────────────────────────────────────┐
│           📚 一起来投票选下一本要读的书           │
│           （副标题：每人限投一票）                 │
├─────────────────────────────────────────────┤
│  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │  封面     │  │  封面     │  │  封面     │   │
│  │  书名 1   │  │  书名 2   │  │  书名 3   │   │
│  │  作者     │  │  作者     │  │  作者     │   │
│  │  简介     │  │  简介     │  │  简介     │   │
│  │ 🗳 12 票  │  │ 🗳 8 票   │  │ 🗳 5 票   │   │
│  │ [投一票]  │  │ [投一票]  │  │ [投一票]  │   │
│  └──────────┘  └──────────┘  └──────────┘   │
│  ┌──────────┐  ┌──────────┐                  │
│  │  书 4     │  │  书 5     │                  │
│  └──────────┘  └──────────┘                  │
├─────────────────────────────────────────────┤
│          当前总票数：35   当前领先：书名 1        │
└─────────────────────────────────────────────┘
```

### 5.2 关键交互流程

**流程 A：用户打开页面**

```
浏览器 ──GET /api/books──▶ 服务器
浏览器 ◀──返回 5 本书的数据 + 票数── 服务器
浏览器：渲染 5 张卡片
```

**流程 B：用户点击"投一票"**

```
用户点击按钮
  │
  ▼
浏览器先查 localStorage：我投过没？
  │
  ├─ 投过了 ──▶ 提示"你已经投过了"，按钮置灰
  │
  └─ 没投过 ──▶ POST /api/vote { bookId: 2 } ──▶ 服务器
                                                    │
                                                    ▼
                                              读 books.json
                                              对应书的票数 +1
                                              写回 books.json
                                                    │
                       浏览器 ◀── 返回最新票数 ──────┘
                          │
                          ▼
                      更新页面上的票数
                      写入 localStorage：voted = true
                      按钮置灰
```

## 6. 数据结构设计

### 6.1 `books.json`（服务器上的"账本"）

```json
[
  { "id": 1, "title": "小王子",        "author": "圣埃克苏佩里", "desc": "一本关于成长和爱的童话",     "cover": "/covers/1.jpg", "votes": 0 },
  { "id": 2, "title": "三体",          "author": "刘慈欣",       "desc": "硬核中国科幻代表作",         "cover": "/covers/2.jpg", "votes": 0 },
  { "id": 3, "title": "活着",          "author": "余华",         "desc": "一个普通人一生的故事",       "cover": "/covers/3.jpg", "votes": 0 },
  { "id": 4, "title": "百年孤独",      "author": "马尔克斯",     "desc": "魔幻现实主义经典",           "cover": "/covers/4.jpg", "votes": 0 },
  { "id": 5, "title": "人类简史",      "author": "赫拉利",       "desc": "从认知革命到未来",           "cover": "/covers/5.jpg", "votes": 0 }
]
```

字段说明：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | 整数 | 书的唯一编号，投票时用它定位 |
| `title` | 字符串 | 书名 |
| `author` | 字符串 | 作者 |
| `desc` | 字符串 | 一句话简介 |
| `cover` | 字符串 | 封面图片路径（也可以先用 emoji 占位） |
| `votes` | 整数 | 当前票数，**会被服务器修改** |

### 6.2 浏览器端 `localStorage`

| Key | Value | 说明 |
|---|---|---|
| `hasVoted` | `"true"` / 不存在 | 标记本浏览器是否已经投过票 |
| `votedBookId` | 数字字符串 | 投给了哪本书（用于显示"你投了 XX"） |

## 7. 接口设计（前后端怎么对话）

> 用最朴素的 RESTful 风格，只有 2 个接口。

### 7.1 获取书单和当前票数

```
GET /api/books
```

**响应示例**：

```json
{
  "ok": true,
  "data": [
    { "id": 1, "title": "小王子", "author": "...", "desc": "...", "cover": "...", "votes": 12 },
    { "id": 2, "title": "三体",   "author": "...", "desc": "...", "cover": "...", "votes": 8  }
  ]
}
```

### 7.2 给某本书投一票

```
POST /api/vote
Content-Type: application/json

{ "bookId": 2 }
```

**成功响应**：

```json
{ "ok": true, "bookId": 2, "votes": 9 }
```

**失败响应**（比如 bookId 不存在）：

```json
{ "ok": false, "error": "book not found" }
```

> **课堂讲解点**：
> - `GET` = "我来取东西"，`POST` = "我来交东西"，对应"点单"和"取奶茶"两种动作。
> - 返回的是 **JSON 格式**，不是 HTML——服务器只给数据，**页面长什么样由浏览器决定**。

## 8. 实施步骤（建议学员用 AI Coding 分 5 步完成）

> 每一步都让学员**先运行、再继续**，确保每一步都能看见效果。
> 建议项目目录结构：
> ```
> book-vote/
> ├── server.js          # 后端入口
> ├── books.json         # 数据"账本"
> ├── package.json       # npm 配置
> └── public/
>     ├── index.html     # 前端页面
>     ├── style.css
>     └── app.js         # 前端脚本
> ```

### Step 1：搭最简 Express 服务器（10 分钟）

- 目标：浏览器打开 `http://127.0.0.1:3000` 能看到 "Hello, 投票！"
- 产出：`package.json` + `server.js`（约 10 行）
- 课前命令：`npm init -y` 然后 `npm install express`
- 给 AI 的提示词建议："帮我用 Node.js + Express 写一个最简单的服务器，监听 3000 端口，访问根路径返回一句中文问候。"

### Step 2：准备书单数据并暴露 `/api/books` 接口（15 分钟）

- 目标：浏览器访问 `/api/books` 能看到 5 本书的 JSON 数据
- 产出：`books.json` + `server.js` 新增一个路由（用 `fs.readFileSync` 读文件）
- **教学重点**：让学员手动改一下 `books.json` 里的票数，刷新页面看到变化，**亲眼看到"服务器从文件读数据"**。

### Step 3：写前端页面，展示 5 本书（20 分钟）

- 目标：页面打开后，能从后端拿到数据并渲染出 5 张书卡
- 产出：`public/index.html` + `public/app.js`（在 `server.js` 里用 `app.use(express.static('public'))` 托管静态资源）
- **教学重点**：打开浏览器 F12 → Network 面板，看到 `/api/books` 这次请求，**让"前后端通信"变得可视化**。

### Step 4：实现投票接口和按钮交互（20 分钟）

- 目标：点击按钮票数 +1，并写回 `books.json`
- 产出：`server.js` 新增 `POST /api/vote` 路由（用 `express.json()` 解析请求体，用 `fs.writeFileSync` 写回文件）+ 前端按钮事件
- **教学重点**：投票后**打开 `books.json` 给学员看**——"看，数据真的被改了，下次别人打开还在！"

### Step 5：加上"每人限投一票"的本地限制（10 分钟）

- 目标：投过票后按钮变灰，刷新后仍然变灰
- 产出：前端使用 `localStorage`
- **教学重点**：提问"如果我清掉浏览器缓存能再投吗？"——引出**"真正的用户身份需要登录系统"**这个话题，作为下一课伏笔。

## 9. 可选拓展（讲完主线后可以挑战）

按难度由低到高：

1. **加排行榜**：在页面底部按票数排序显示 Top 3。
2. **加投票动画**：点击后票数数字滚动 +1。
3. **加管理页面**：`/admin` 路由可以清空所有票数（讲到这里可以引出"权限"概念）。
4. **加 IP 限制**：服务器记录每个 IP 投过谁，更严格地防重复（引出"为什么 IP 也不完全可靠"）。
5. **拆分数据文件**：把投票记录单独存到 `votes.json`，和书目数据 `books.json` 分开，体会"数据分表"的思路（不引入数据库，仍然只用文件）。
6. **部署上线**：用 Replit 或 Render 部署，让同学们用真实网址互相投票。

## 11. 给讲师的备忘

- 准备好一台能让大家同时访问的电脑（或用内网穿透），让学员**真切感受到"我投的票别人也看到了"**——这是本节课最重要的"哇时刻"。
- 提前把 5 本书的封面图放好，避免课上学员卡在素材上。
- 强调"防重复投票只是本地限制"，不要让学员误以为这就是安全方案。
- **关于文件存储的小坑**：用 JSON 文件当账本，多人同时点击投票时理论上可能出现"同时读写"导致票数丢失。课堂演示规模（十几个人）一般不会出问题；如果学员追问，可作为思考题——"这就是后面学数据库的动机之一"，但**本课不引入数据库**。
- AI Coding 提示词要**鼓励学员自己写**，讲师可以提供示例但不要让学员只会复制粘贴。
