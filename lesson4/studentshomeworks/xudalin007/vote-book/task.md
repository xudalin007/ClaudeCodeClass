# 图书投票应用任务管理计划

## 1. 项目目标概述

本项目要实现一个网页版本的图书投票应用。MVP 阶段支持展示约 6 本图书，用户可点击按钮为图书投票，票数由 Node.js 后端处理并存储到本地 JSON 文件中。

项目当前目标是在已完成投票 MVP、认证、投票限制、后台基础能力、找回密码、本地图书元数据补全、联网抓取图书元数据和已有图书真实封面刷新能力的基础上，保持 `spec.md`、`task.md` 与实际代码和数据状态一致。

## 2. MVP、认证与后台扩展开发范围

### 2.1 投票 MVP 已包含范围

- 展示图书列表。
- 展示每本书的基础信息和当前票数。
- 支持用户点击按钮投票。
- 投票成功后更新对应图书票数。
- 后端提供图书列表接口和投票接口。
- 使用 `book.json` 存储图书信息。
- 使用 `vote.json` 存储投票信息。
- 提供基础错误处理和用户提示。
- 使用文件读写方案保证 MVP 阶段的数据可持久化。

### 2.2 基础认证扩展包含范围

- 新增 `user.json` 保存用户信息。
- 支持用户注册。
- 支持用户登录。
- 支持用户退出登录。
- 支持获取当前登录用户信息。
- 前端展示未登录状态、注册/登录入口、已登录用户名和退出入口。
- 密码使用 Node.js 内置 `crypto` 进行哈希和盐处理，不明文保存。
- 登录态使用 Cookie + 服务端内存 Session。
- 未登录用户仍可浏览图书列表，但不能投票。
- 所有已登录用户，包括管理员，都可以自行修改自己的登录密码。
- 所有用户都可以通过找回问题和找回答案重置自己的密码。
- 找回密码不返回旧密码，只允许设置新密码。
- 找回答案使用独立随机盐和哈希保存，不保存明文。
- 用户可以绑定 E-mail，用于接收找回密码验证码。
- 邮箱验证码找回密码最多允许输错 3 次，第 3 次错误后验证码失效。
- 邮箱验证码不保存明文，只保存哈希、盐、过期时间、发送时间和错误次数。

### 2.3 投票限制扩展包含范围

- 只允许已登录用户投票。
- 未登录用户点击投票时提示先登录。
- 每个用户最多累计投 3 票。
- 3 票可以投给同一本书，也可以投给不同图书。
- 用户投满 3 票后，后端拒绝继续投票并返回明确错误。
- 前端展示当前用户已用票数或剩余票数。
- `vote.json` 需要保存每个用户的投票记录，用于判断投票次数。

### 2.4 管理员后台扩展包含范围

- 支持管理员角色。
- 管理员可以访问后台页面。
- 管理员可以查看图书列表。
- 管理员可以修改已有图书的书名、作者、简介和封面地址。
- 管理员可以新增图书，从而增加候选图书本数。
- 管理员新增图书时只需要填写书名和作者。
- 管理员新增图书时，系统必须优先联网从公开图书信息源抓取真实封面和真实图书简介。
- 管理员新增图书时可以手动填写简介和封面地址，手动内容优先于联网抓取结果。
- 联网抓取失败时，系统才使用本地匹配元数据、默认封面和基础简介兜底。
- 管理员可以批量刷新已有候选图书真实封面，将本地生成封面替换为联网抓取并缓存的真实封面。
- 管理员可以删除图书，从而减少候选图书本数。
- 管理员新增或删除图书时，需要同步维护 `book.json` 和 `vote.json`。
- 普通用户不能访问后台页面和后台接口。
- 管理员可以发起新一轮投票，清空票数和用户投票记录。

### 2.5 不包含范围

- 反作弊机制。
- 邮箱验证、手机号验证、短信找回密码。
- 第三方登录。
- Session 持久化；服务重启后允许登录态失效。
- 数据库接入。
- 图书分类、排序、标签等高级字段维护。
- 独立排行榜页面。
- 评论、收藏、分享等社交功能。

## 7. 后续同步进度更新规则

后续开发过程中，应持续更新本文件，确保任务状态与实际进度一致。

### 7.1 状态更新规则

- 开始某项任务时，将状态从 `todo` 改为 `in-progress`。
- 完成某项任务并通过验收后，将状态改为 `done`。
- 如果任务实现完成但未验证，仍保持 `in-progress`。
- 如果发现任务拆分不合理，可以新增、拆分或合并任务，但需保持验收标准清晰。

### 7.2 进度摘要更新规则

每次完成一个阶段后，应更新“当前项目进度摘要”：

- 说明已完成的阶段。
- 说明正在进行的阶段。
- 说明下一步计划。
- 记录发现的风险、阻塞或范围变化。

### 7.3 验收记录规则

每次完成关键能力后，建议记录验证方式：

- 接口验证：记录请求路径、请求方法和期望响应。
- 页面验证：记录用户操作路径和预期页面反馈。
- 文件验证：记录 `book.json`、`vote.json`、`user.json` 是否被正确读取或写入。

### 7.4 范围控制规则

后续开发期间应优先完成 `spec.md` 中定义的核心能力。以下需求暂不进入当前开发范围：

- 数据库迁移。
- 邮箱、短信找回密码和第三方登录。
- 复杂 UI 动效。

如需加入新功能，应先更新 `spec.md`，再同步更新 `task.md`。
