# 图书投票应用 MVP、用户认证与后台管理扩展规格文档

## 1. 产品目标和当前范围

本项目目标是实现一个网页版本的图书投票应用，用于展示约 6 本候选图书，并允许访问者为喜欢的图书投票。

投票 MVP 阶段重点验证以下能力：

- 展示图书列表，包括书名、作者、简介、封面等基础信息。
- 展示每本书当前获得的票数。
- 用户点击“投票”按钮后，对应图书票数更新。
- 后端使用 Node.js 提供数据读取和投票接口。
- 数据使用本地 JSON 文件存储，不引入数据库。

当前已在 MVP 基础上增加基础用户认证能力：

- 支持用户注册、登录、退出登录。
- 支持前端检查当前登录状态并展示当前用户名。
- 支持已登录用户修改自己的密码。
- 支持用户通过找回凭证重置自己的密码；找回密码不返回旧密码。
- 用户数据继续使用本地 JSON 文件存储，不引入数据库。
- 密码不能明文存储，应使用 Node.js 内置 `crypto` 做哈希和盐处理。
- 找回密码答案不能明文存储，应使用独立随机盐和哈希保存。
- 登录态使用简单 Cookie + 服务端内存 Session 实现。
- 未登录用户只能浏览图书列表，不能投票。

下一阶段在现有基础上增加投票权限、投票次数限制和管理员后台能力：

- 只允许已登录用户投票。
- 每个用户最多可投 3 票，3 票可以投给同一本书，也可以投给不同图书。
- 用户投满 3 票后，再次投票应被拒绝，并返回明确错误。
- 管理员可以在后台修改图书名称、作者、简介和封面地址。
- 管理员可以在后台调整图书本数，包括新增图书和删除图书。
- 管理员新增图书时只需要填写书名和作者，系统必须联网抓取真实封面和图书简介；管理员也可以手动补充或覆盖封面和简介。
- 普通用户不能访问后台管理能力。

## 2. 主要功能列表

### 2.1 图书列表展示

- 页面展示约 6 本图书。
- 每本图书以卡片形式展示基础信息。
- 每张卡片包含：
  - 图书名称
  - 作者
  - 图书简介
  - 封面图片地址
  - 当前票数
  - 投票按钮
- 当前候选图书必须优先使用联网抓取并缓存的真实图书封面，不应使用 AI 生成封面作为最终封面。
- 当前候选图书必须使用可见的图书封面资产，不应全部指向默认占位图。
- 默认封面 `/covers/default.svg` 只作为新增图书抓取失败时的兜底，不作为已有候选图书的常态封面。
- 候选图书卡片在桌面宽度下优先使用 3 列以内的网格布局，避免卡片过窄。
- 封面必须限制在封面容器内完整显示，不能溢出并覆盖书名、作者、简介或投票区域。
- 书名和简介应限制展示行数，避免单本图书文本过长破坏卡片高度。

### 2.2 投票功能

- 未登录用户点击“投票”按钮时，前端应提示先登录。
- 用户点击某本书的“投票”按钮后，前端向后端发送投票请求。
- 后端必须校验当前请求是否来自已登录用户。
- 后端校验图书是否存在。
- 后端校验当前用户累计投票次数是否小于 3。
- 校验通过后，对应图书票数增加 1。
- 后端将新的投票数据写入 `vote.json`。
- 前端刷新对应图书票数，或重新拉取最新列表数据。
- 如果用户已投满 3 票，后端返回错误，前端提示“每个用户最多可投 3 票”。

### 2.3 票数展示

- 页面加载时显示每本书当前票数。
- 投票成功后，页面上的票数应立即更新。
- 如果投票失败，应展示明确的错误提示。

### 2.4 基础错误处理

- 图书数据读取失败时，页面展示加载失败提示。
- 投票请求失败时，页面提示用户稍后重试。
- 后端返回统一的错误信息结构，便于前端处理。

### 2.5 基础用户注册与登录

- 用户可以通过用户名和密码注册新账号。
- 注册时后端校验用户名是否重复。
- 注册成功后可选择直接登录，或提示用户手动登录。
- 已注册用户可以通过用户名和密码登录。
- 登录成功后后端创建 Session，并通过 Cookie 保存会话标识。
- 前端可以通过接口获取当前登录用户信息。
- 登录后页面展示当前用户名和“退出登录”入口。
- 用户可以退出登录，后端销毁当前 Session，前端恢复未登录状态。
- 未登录用户仍可浏览图书列表，但不能投票。
- 普通用户登录后可以投票，但不能访问管理员后台。
- 所有已登录用户，包括普通用户和管理员，都可以自行修改自己的登录密码。
- 修改密码时必须输入当前密码和新密码。
- 修改成功后，后端更新 `user.json` 中当前用户的 `passwordHash` 和 `passwordSalt`。
- 修改密码不改变用户 `id`、`username`、`role` 和既有投票记录。
- 所有用户都可以通过找回密码流程重置自己的密码。
- 找回密码流程不展示、不返回旧密码，只允许用户设置新密码。
- 找回密码优先支持 E-mail 接收验证码；验证码通过后允许用户设置新密码。
- 邮箱验证码错误次数限制为 3 次，超过后验证码失效，需要重新发送。
- 历史的“找回问题 + 找回答案”能力可以保留作为已实现的基础找回方式，但后续主流程应切换为邮箱验证码。
- 注册新用户时应填写找回问题和找回答案。
- 注册新用户时应填写 E-mail，用于接收找回密码验证码。
- 已登录用户可以设置或更新自己的找回问题和找回答案。
- 已登录用户可以设置或更新自己的 E-mail。
- 找回答案必须使用独立盐和哈希存储，不允许明文保存。
- 邮箱验证码不能明文持久化保存，应保存哈希、过期时间、错误次数和发送时间。

### 2.6 管理员后台图书管理

- 管理员可以访问后台页面。
- 管理员可以查看当前图书列表。
- 管理员可以修改已有图书的书名、作者、简介和封面地址。
- 管理员可以新增图书，从而增加候选图书本数。
- 管理员新增图书时必须填写书名和作者。
- 管理员新增图书时，系统应根据书名和作者从公开图书信息源联网抓取真实封面图片地址和图书简介。
- 管理员新增图书时也可以手动填写简介和封面地址；手动填写内容优先于联网抓取结果。
- 联网抓取失败时，后端应使用本地匹配封面或默认封面、基础简介兜底，并向前端返回明确提示，方便管理员后续手动修正。
- 管理员可以对已有候选图书发起真实封面刷新，将本地生成封面替换为联网抓取并缓存的真实封面。
- 管理员可以删除图书，从而减少候选图书本数。
- 管理员新增图书时，应同步初始化该图书的票数。
- 管理员删除图书时，应同步处理该图书的票数和相关投票记录，避免前端展示无效数据。
- 普通用户访问后台接口时，应返回权限不足错误。

### 2.7 管理员发起新一轮投票

- 管理员可以在后台发起新一轮投票。
- 新一轮投票会清空所有图书当前票数。
- 新一轮投票会清空所有用户在上一轮的投票记录。
- 新一轮投票开始后，每个用户重新获得 3 票。
- 新一轮投票不删除图书列表，不删除用户账号，不改变用户角色。
- 发起新一轮投票属于高风险操作，前端必须二次确认。
- 发起成功后，前端应刷新图书票数、当前用户剩余票数和后台数据。

## 3. 不包含的功能范围

当前阶段明确不包含以下功能：

- IP、Cookie、设备指纹等反作弊机制。
- 手机号验证、短信找回密码。
- 第三方登录，例如微信、GitHub、Google。
- Session 持久化到文件或数据库；服务重启后允许登录态失效。
- 数据库持久化。
- 图书分类、排序、标签等高级字段维护。
- 排行榜单独页面。
- 评论、收藏、分享等社交功能。
- 复杂动画和高级 UI 交互。

这些能力可在 MVP 验证完成后作为后续迭代方向。

## 4. 前端页面设计思路

前端可以采用单页应用或普通静态页面实现。MVP 阶段建议优先保持结构简单。

### 4.1 页面布局

页面建议包含以下区域：

- 顶部标题区：展示应用名称，例如“图书投票”。
- 用户状态区：展示未登录状态、登录/注册入口，或当前登录用户名、剩余票数和退出按钮。
- 管理入口区：仅管理员登录后展示后台入口。
- 说明区：简要说明用户可以为喜欢的图书投票。
- 图书列表区：以网格或列表形式展示所有图书卡片。
- 状态提示区：展示加载中、投票成功、投票失败等反馈。

### 4.2 图书卡片设计

每本书使用独立卡片展示，卡片内容包括：

- 封面图片。
- 书名和作者。
- 简短介绍。
- 当前票数。
- “投票”按钮。

### 4.3 交互设计

- 页面初始化时请求后端接口获取图书和票数数据。
- 页面初始化时请求 `GET /api/auth/me` 检查当前登录状态。
- 未登录用户点击投票按钮时，应提示先登录，不发送投票请求。
- 已登录用户点击投票按钮后，按钮可以进入短暂禁用状态，避免用户连续快速点击造成重复请求。
- 投票成功后更新票数，并提示“投票成功”。
- 投票成功后更新当前用户剩余可投票次数。
- 用户已投满 3 票时，应禁用或弱化投票入口，并提示投票次数已用完。
- 投票失败后恢复按钮状态，并提示失败原因。

### 4.4 认证交互设计

- 未登录状态下，页面展示“注册”和“登录”入口。
- 注册表单包含用户名、E-mail、密码、确认密码、找回问题和找回答案。
- 登录表单包含用户名和密码。
- 登录区域提供“忘记密码”入口。
- 所有密码输入框旁边应提供“显示/隐藏密码”切换按钮。
- 用户点击“显示密码”后，当前密码输入框内容以明文显示；再次点击后恢复隐藏。
- 密码可见切换只影响前端输入框展示，不改变后端密码哈希和存储规则。
- 前端应做基础输入校验，例如用户名不能为空、E-mail 格式合法、密码不能为空、两次密码输入一致、找回问题和找回答案不能为空。
- 注册或登录请求进行中，提交按钮应短暂禁用。
- 注册成功、登录成功、退出登录失败等状态应给出明确提示。
- 登录后页面显示当前用户名，并隐藏注册/登录入口。
- 退出登录成功后，页面恢复未登录状态。
- 登录后页面提供“修改密码”入口。
- 修改密码表单包含当前密码、新密码、确认新密码。
- 修改密码表单中的所有密码输入框都应支持显示/隐藏切换。
- 修改密码成功后应清空表单，并提示用户修改成功。
- 修改密码失败时，应展示当前密码错误、新密码格式不合法等明确错误。
- 登录后页面提供“设置找回密码凭证”入口。
- 找回凭证表单包含找回问题、找回答案，并支持更新当前用户的找回凭证。
- 忘记密码表单先输入用户名并获取找回问题，再填写找回答案、新密码和确认新密码。
- 找回密码成功后应提示用户使用新密码重新登录。
- 忘记密码表单优先使用邮箱验证码流程：输入要重置的用户名、任意接收验证码的 E-mail，请求验证码，填写验证码、新密码和确认新密码。
- 邮箱验证码发送成功后应提示用户检查邮箱，并展示验证码有效期和剩余错误次数规则。
- 邮箱验证码错误 3 次后应提示验证码已失效，需要重新发送。
- 找回密码失败时，应展示用户不存在、接收邮箱格式不合法、验证码错误、验证码过期、错误次数超限、新密码格式不合法等明确错误。

### 4.5 管理员后台页面设计

- 后台页面只对管理员展示。
- 后台页面应提供图书列表管理区域。
- 每本图书至少展示 `id`、当前书名、作者、简介、封面地址和当前票数。
- 管理员可以修改书名、作者、简介、封面地址并保存。
- 管理员可以新增图书，新增时必须填写书名和作者。
- 新增图书表单中只有书名和作者是必填项；简介和封面地址为可选项。
- 简介和封面地址为空时，后端必须联网抓取真实封面和图书简介并补全。
- 新增图书时前端应展示“正在联网抓取真实封面和简介”的状态提示。
- 新增图书成功后，如果联网抓取失败并使用了兜底信息，前端应提示管理员可在后台继续编辑。
- 后台应提供“刷新真实封面”入口，用于批量处理已有图书的本地生成封面；该操作只更新封面地址，不自动覆盖管理员已维护的简介。
- 管理员可以删除图书，删除前应二次确认。
- 后台操作成功后，应刷新图书列表并给出明确提示。
- 后台操作失败时，应显示权限不足、数据校验失败或文件写入失败等错误信息。
- 后台页面应提供“发起新一轮投票”入口。
- 发起新一轮投票前，前端必须展示二次确认，说明会清空票数和用户投票记录。
- 发起成功后，前台和后台都应展示所有图书票数归零，当前用户剩余票数恢复为 3。

## 5. 后端接口设计思路

后端使用 Node.js 实现，负责读取本地 JSON 文件、处理投票请求、处理用户认证请求并返回数据。

### 5.1 获取图书列表接口

- 方法：`GET`
- 路径：`/api/books`
- 功能：返回图书信息和对应票数。
- 数据来源：
  - 从 `book.json` 读取图书基础信息。
  - 从 `vote.json` 读取投票信息。
  - 在后端合并后返回给前端。

推荐响应结构：

```json
{
  "success": true,
  "data": [
    {
      "id": "book-001",
      "title": "示例图书",
      "author": "示例作者",
      "description": "图书简介",
      "coverUrl": "/images/book-001.jpg",
      "votes": 10
    }
  ]
}
```

### 5.2 投票接口

- 方法：`POST`
- 路径：`/api/votes`
- 功能：为指定图书增加一票。
- 登录要求：必须登录。
- 投票次数限制：每个用户最多累计投 3 票。

推荐请求结构：

```json
{
  "bookId": "book-001"
}
```

推荐响应结构：

```json
{
  "success": true,
  "data": {
    "bookId": "book-001",
    "votes": 11,
    "userVoteCount": 1,
    "remainingVotes": 2
  }
}
```

错误场景：

- 用户未登录，返回 `UNAUTHORIZED`。
- 图书不存在，返回 `BOOK_NOT_FOUND`。
- 用户已投满 3 票，返回 `VOTE_LIMIT_REACHED`。
- 投票数据写入失败，返回 `DATA_WRITE_ERROR`。

### 5.3 用户注册接口

- 方法：`POST`
- 路径：`/api/auth/register`
- 功能：创建新用户。

推荐请求结构：

```json
{
  "username": "demo",
  "email": "demo@example.com",
  "password": "password123",
  "recoveryQuestion": "我最喜欢的一本书是什么？",
  "recoveryAnswer": "代码整洁之道"
}
```

推荐响应结构：

```json
{
  "success": true,
  "data": {
    "id": "user-001",
    "username": "demo",
    "role": "user"
  }
}
```

错误场景：

- 用户名为空或格式不合法，返回 `INVALID_USERNAME`。
- 邮箱为空或格式不合法，返回 `INVALID_EMAIL`。
- 密码为空或长度不足，返回 `INVALID_PASSWORD`。
- 找回问题为空或格式不合法，返回 `INVALID_RECOVERY_QUESTION`。
- 找回答案为空或格式不合法，返回 `INVALID_RECOVERY_ANSWER`。
- 用户名已存在，返回 `USERNAME_EXISTS`。
- 邮箱已被其他用户使用，返回 `EMAIL_EXISTS`。
- 用户数据写入失败，返回 `USER_WRITE_ERROR`。

### 5.4 用户登录接口

- 方法：`POST`
- 路径：`/api/auth/login`
- 功能：校验用户名和密码，创建登录 Session。

推荐请求结构：

```json
{
  "username": "demo",
  "password": "password123"
}
```

推荐响应结构：

```json
{
  "success": true,
  "data": {
    "id": "user-001",
    "username": "demo",
    "role": "user"
  }
}
```

登录成功后，后端应通过 `Set-Cookie` 写入 Session Cookie。

错误场景：

- 用户不存在或密码错误，返回 `INVALID_CREDENTIALS`。
- 用户数据读取失败，返回 `USER_READ_ERROR`。

### 5.5 用户退出登录接口

- 方法：`POST`
- 路径：`/api/auth/logout`
- 功能：销毁当前 Session 并清除 Cookie。

推荐响应结构：


### 5.6.4 发送邮箱找回密码验证码接口

- 方法：`POST`
- 路径：`/api/auth/password-reset-code`
- 功能：用户输入要重置的用户名和任意接收验证码的 E-mail 后，后端向该 E-mail 发送找回密码验证码。
- 登录要求：不需要登录。

推荐请求结构：

```json
{
  "account": "demo",
  "email": "receiver@example.com"
}
```

推荐响应结构：

```json
{
  "success": true,
  "data": {
    "codeSent": true,
    "expiresInSeconds": 600,
    "retryAfterSeconds": 60,
    "maxAttempts": 3
  }
}
```

处理规则：

- `account` 表示要重置密码的账号，必须是用户名；该规则适用于普通用户和管理员。
- `email` 表示本次接收验证码的邮箱，可以是任意合法邮箱，不要求与用户资料中的 E-mail 一致。
- 发送验证码时，不需要用户已绑定邮箱。
- 后端应记录本次验证码对应的接收邮箱，用于后续重置时校验验证码归属。
- 验证码建议为 6 位数字。
- 验证码有效期建议为 10 分钟。
- 同一用户 60 秒内不允许重复发送验证码。
- 后端不持久化保存验证码明文，只保存验证码哈希、盐、过期时间、发送时间和错误次数。
- 邮件发送失败时，不应写入可用验证码。
- 安全取舍：该流程允许任何人向任意邮箱请求某账号的验证码；只要能拿到验证码即可重置该账号密码。MVP 按用户当前要求实现，后续生产化建议增加身份校验、管理员限制、审计日志或人工审核。

错误场景：

- 用户名为空或格式不合法，返回 `INVALID_ACCOUNT`。
- 接收邮箱为空或格式不合法，返回 `INVALID_EMAIL`。
- 用户不存在，返回 `USER_NOT_FOUND`。
- 发送过于频繁，返回 `PASSWORD_RESET_CODE_TOO_FREQUENT`。
- 邮件服务未配置，返回 `EMAIL_SERVICE_NOT_CONFIGURED`。
- 邮件发送失败，返回 `EMAIL_SEND_FAILED`。
- 用户数据写入失败，返回 `USER_WRITE_ERROR`。

### 5.6.5 通过邮箱验证码重置密码接口

- 方法：`POST`
- 路径：`/api/auth/reset-password-by-code`
- 功能：用户通过邮箱验证码重置自己的登录密码。
- 登录要求：不需要登录。

推荐请求结构：

```json
{
  "account": "demo",
  "email": "receiver@example.com",
  "code": "123456",
  "newPassword": "newPassword123"
}
```

推荐响应结构：

```json
{
  "success": true,
  "data": {
    "passwordReset": true
  }
}
```

错误次数规则：

- 同一个验证码最多允许输错 3 次。
- 第 1 次和第 2 次输错时，返回 `INVALID_PASSWORD_RESET_CODE`，并返回剩余可尝试次数。
- 第 3 次输错时，验证码立即失效，返回 `PASSWORD_RESET_CODE_LOCKED`。
- 验证码过期后返回 `PASSWORD_RESET_CODE_EXPIRED`。
- 验证成功或验证码失效后，应清理该用户当前验证码哈希、盐、过期时间和错误次数。

错误场景：

- 用户名为空或格式不合法，返回 `INVALID_ACCOUNT`。
- 接收邮箱为空或格式不合法，返回 `INVALID_EMAIL`。
- 接收邮箱与验证码发送时记录的邮箱不一致，返回 `INVALID_PASSWORD_RESET_EMAIL`。
- 用户不存在，返回 `USER_NOT_FOUND`。
- 验证码为空或格式不合法，返回 `INVALID_PASSWORD_RESET_CODE`。
- 验证码未发送或已失效，返回 `PASSWORD_RESET_CODE_NOT_FOUND`。
- 验证码错误，返回 `INVALID_PASSWORD_RESET_CODE`。
- 验证码错误次数达到 3 次，返回 `PASSWORD_RESET_CODE_LOCKED`。
- 验证码过期，返回 `PASSWORD_RESET_CODE_EXPIRED`。
- 新密码为空或长度不足，返回 `INVALID_PASSWORD`。
- 用户数据写入失败，返回 `USER_WRITE_ERROR`。

### 5.9 管理员新增图书接口

- 方法：`POST`
- 路径：`/api/admin/books`
- 功能：新增一本候选图书，用于增加候选图书本数。
- 权限要求：必须登录且用户角色为 `admin`。
- 管理员必须提供书名和作者。
- 简介和封面地址可以手动提供；未提供时后端必须联网抓取真实封面和图书简介并补全。
- 手动提供的简介和封面地址优先于联网抓取结果。

推荐请求结构：

```json
{
  "title": "新图书名称",
  "author": "新图书作者"
}
```

可选手动覆盖字段：

```json
{
  "title": "新图书名称",
  "author": "新图书作者",
  "description": "",
  "coverUrl": ""
}
```

推荐响应结构：

```json
{
  "success": true,
  "data": {
    "id": "book-007",
    "title": "新图书名称",
    "author": "新图书作者",
    "description": "新图书简介",
    "coverUrl": "/covers/new-book.svg",
    "votes": 0,
    "metadata": {
      "descriptionFetched": true,
      "coverFetched": true,
      "source": "open-library",
      "confidence": "high",
      "fallbackUsed": false,
      "warnings": []
    }
  }
}
```

联网抓取规则：

- 后端根据 `title` 和 `author` 调用公开图书信息源，联网尝试获取真实简介和真实封面。
- 推荐优先使用无需登录即可访问且当前环境可连通的数据源，例如 Douban 搜索结构化数据；Google Books、Open Library 等公开 API 可作为候选补充源，具体数据源应封装为可替换模块。
- 抓取逻辑应优先匹配书名和作者均一致的结果；如只能匹配到书名，应在响应 `metadata.warnings` 中提示匹配可信度较低。
- 抓取到的封面地址必须是可被浏览器直接访问的图片 URL，或后端已下载/缓存到本地静态目录后的 URL。
- 如果管理员手动填写 `description`，后端不覆盖该简介。
- 如果管理员手动填写 `coverUrl`，后端不覆盖该封面地址。
- 如果联网抓取失败且管理员未手动填写简介，后端生成基础简介，例如“《书名》是作者创作的一本候选图书。”。
- 如果联网抓取失败且管理员未手动填写封面，后端优先使用本地匹配封面；仍无匹配时使用默认封面 `/covers/default.svg`。
- 后端应在响应 `metadata.warnings` 中返回抓取失败、兜底使用等提示。

## 6. `book.json`、`vote.json` 和 `user.json` 推荐数据结构

### 6.1 `book.json`

`book.json` 用于保存图书基础信息，不保存票数。

推荐结构：

```json
[
  {
    "id": "book-001",
    "title": "示例图书 1",
    "author": "示例作者 1",
    "description": "这是一本示例图书。",
    "coverUrl": "/images/book-001.jpg"
  },
  {
    "id": "book-002",
    "title": "示例图书 2",
    "author": "示例作者 2",
    "description": "这是另一本示例图书。",
    "coverUrl": "/images/book-002.jpg"
  }
]
```

字段说明：

- `id`：图书唯一标识，投票时使用。
- `title`：图书名称，前台展示，后台新增和编辑时必填。
- `author`：作者，前台展示，后台新增和编辑时必填。
- `description`：图书简介，前台展示，后台编辑时必填；后台新增时可由系统联网抓取或兜底生成。
- `coverUrl`：封面图片地址，前台展示，后台编辑时必填；后台新增时可由系统联网抓取或使用本地匹配封面、默认封面。
- 已有候选图书的 `coverUrl` 应优先指向真实封面缓存路径，例如 `/covers/fetched/book-cover-xxx.jpg`。
- `/covers/book-*.svg` 只能作为历史本地生成封面或兜底临时封面，不应作为已有候选图书的最终封面。
- 本地缓存封面建议使用竖版比例，确保在图书卡片和后台预览中能完整显示。

### 6.2 `vote.json`

`vote.json` 用于保存每本书的票数和每个用户的投票记录。

推荐结构：

```json
{
  "bookVotes": {
    "book-001": 10,
    "book-002": 5,
    "book-003": 0,
    "book-004": 3,
    "book-005": 8,
    "book-006": 1
  },
  "userVotes": {
    "user-001": [
      {
        "bookId": "book-001",
        "votedAt": "2026-05-12T09:00:00.000Z"
      },
      {
        "bookId": "book-001",
        "votedAt": "2026-05-12T09:05:00.000Z"
      },
      {
        "bookId": "book-002",
        "votedAt": "2026-05-12T09:10:00.000Z"
      }
    ]
  }
}
```

结构说明：

- `bookVotes`：对象的 key 是图书 `id`，value 是对应图书当前票数。
- `userVotes`：对象的 key 是用户 `id`，value 是该用户的投票记录数组。
- 每个用户的 `userVotes[userId]` 数组长度最多为 `3`。
- 同一用户可以把 3 票投给同一本书，也可以投给不同图书。
- 如果某本书在 `bookVotes` 中不存在，后端可以默认其票数为 `0`。
- 如需兼容旧版 `vote.json` 的 `{ "book-001": 0 }` 结构，后端启动时应迁移为新结构。

### 6.3 `user.json`

`user.json` 用于保存用户基础信息和密码哈希信息，不保存明文密码。

推荐结构：

```json
[
  {
    "id": "user-001",
    "username": "demo",
    "email": "demo@example.com",
    "role": "user",
    "passwordHash": "base64-encoded-hash",
    "passwordSalt": "base64-encoded-salt",
    "recoveryQuestion": "我最喜欢的一本书是什么？",
    "recoveryAnswerHash": "base64-encoded-answer-hash",
    "recoveryAnswerSalt": "base64-encoded-answer-salt",
    "recoveryUpdatedAt": "2026-05-12T09:00:00.000Z",
    "passwordResetCodeHash": "base64-encoded-code-hash",
    "passwordResetCodeSalt": "base64-encoded-code-salt",
    "passwordResetCodeExpiresAt": "2026-05-12T09:10:00.000Z",
    "passwordResetCodeSentAt": "2026-05-12T09:00:00.000Z",
    "passwordResetCodeEmail": "receiver@example.com",
    "passwordResetCodeFailedAttempts": 0,
    "createdAt": "2026-05-12T09:00:00.000Z"
  }
]
```

字段说明：

- `id`：用户唯一标识。
- `username`：用户名，注册和登录时使用，应保持唯一。
- `email`：用户邮箱，用于接收找回密码验证码，应保持唯一；历史用户可为空，登录后补充。
- `role`：用户角色，普通用户为 `user`，管理员为 `admin`。
- `passwordHash`：使用 Node.js 内置 `crypto` 生成的密码哈希。
- `passwordSalt`：生成密码哈希时使用的随机盐。
- `recoveryQuestion`：找回密码问题，新注册用户必填；历史用户可登录后补充。
- `recoveryAnswerHash`：找回答案哈希，不保存明文答案。
- `recoveryAnswerSalt`：生成找回答案哈希时使用的随机盐，应与密码盐独立。
- `recoveryUpdatedAt`：找回凭证最近更新时间，建议使用 ISO 8601 字符串。
- `passwordResetCodeHash`：邮箱验证码哈希，不保存验证码明文。
- `passwordResetCodeSalt`：生成验证码哈希时使用的随机盐。
- `passwordResetCodeExpiresAt`：验证码过期时间。
- `passwordResetCodeSentAt`：最近一次验证码发送时间，用于限制发送频率。
- `passwordResetCodeEmail`：本次验证码发送到的接收邮箱，用于重置密码时校验验证码归属；可与用户资料中的 `email` 不一致。
- `passwordResetCodeFailedAttempts`：当前验证码错误次数，最大为 3。
- `createdAt`：用户创建时间，建议使用 ISO 8601 字符串。

管理员处理原则：

- 普通注册用户默认角色为 `user`。
- 管理员账号可以通过手动初始化 `user.json`、启动脚本或后续管理员创建流程生成。
- MVP 阶段不要求普通用户自行升级为管理员。

密码处理原则：

- 不允许在 `user.json` 中保存明文密码。
- 不允许在 `user.json` 中保存明文找回答案。
- 每个用户必须使用独立随机盐。
- 密码和找回答案应分别使用独立随机盐。
- 推荐使用 `crypto.pbkdf2`、`crypto.scrypt` 或同等级别的内置密码哈希方案。
- 登录校验时应使用相同算法重新计算哈希并比较结果。
- 找回密码时应使用相同算法重新计算找回答案哈希并比较结果，比较通过后只重置密码，不返回旧密码。
- 邮箱验证码找回密码时应使用相同算法重新计算验证码哈希并比较结果，比较通过后只重置密码，不返回旧密码。
- 邮箱验证码验证成功、过期或错误次数达到 3 次后，应清理该用户验证码相关字段，防止重复使用。

## 7. 投票与认证流程说明

### 7.1 投票流程

1. 用户打开网页。
2. 前端请求 `GET /api/books`。
3. 前端请求 `GET /api/auth/me`。
4. 后端读取 `book.json` 和 `vote.json`。
5. 后端根据图书 `id` 合并图书信息和票数。
6. 前端渲染图书列表和当前用户剩余票数。
7. 未登录用户点击投票按钮时，前端提示先登录，不发送投票请求。
8. 已登录用户点击某本书的“投票”按钮。
9. 前端发送 `POST /api/votes`，请求体包含 `bookId`。
10. 后端根据 Cookie 校验用户是否登录。
11. 后端检查 `bookId` 是否存在于 `book.json`。
12. 后端读取 `vote.json`，检查当前用户累计投票记录是否少于 3 条。
13. 如果用户已投满 3 票，返回 `VOTE_LIMIT_REACHED`。
14. 如果校验通过，后端将对应图书票数加 1，并追加当前用户投票记录。
15. 后端写回 `vote.json`。
16. 后端返回该图书最新票数、用户已用票数和剩余票数。
17. 前端更新页面上的票数和剩余票数，并提示投票成功。

### 7.2 注册流程

1. 用户填写用户名、密码、确认密码、找回问题和找回答案。
2. 前端做基础输入校验。
3. 前端发送 `POST /api/auth/register`。
4. 后端读取 `user.json`。
5. 后端检查用户名是否已存在。
6. 后端为密码生成随机盐和哈希。
7. 后端为找回答案生成独立随机盐和哈希。
8. 后端将新用户写入 `user.json`。
9. 后端返回用户基础信息，不返回密码哈希、密码盐、找回答案哈希和找回答案盐。
10. 前端提示注册成功。

### 7.3 登录流程

1. 用户填写用户名和密码。
2. 前端发送 `POST /api/auth/login`。
3. 后端读取 `user.json` 并查找用户。
4. 后端使用用户的 `passwordSalt` 重新计算密码哈希。
5. 后端比较计算结果和 `passwordHash`。
6. 校验成功后创建内存 Session。
7. 后端通过 Cookie 返回 Session 标识。
8. 前端更新当前用户状态并展示用户名。

### 7.4 退出登录流程

1. 用户点击“退出登录”。
2. 前端发送 `POST /api/auth/logout`。
3. 后端根据 Cookie 找到当前 Session。
4. 后端删除 Session 并清除 Cookie。
5. 前端恢复未登录状态。

### 7.5 修改当前用户密码流程

1. 已登录用户打开“修改密码”表单。
2. 用户填写当前密码、新密码、确认新密码。
3. 前端校验新密码格式和两次输入是否一致。
4. 前端发送 `POST /api/auth/change-password`。
5. 后端根据 Cookie 校验当前用户是否登录。
6. 后端读取 `user.json` 并找到当前用户。
7. 后端用当前用户的 `passwordSalt` 校验当前密码。
8. 校验失败时返回 `INVALID_CURRENT_PASSWORD`。
9. 校验成功后，为新密码生成新的随机盐和哈希。
10. 后端更新当前用户的 `passwordHash` 和 `passwordSalt`。
11. 后端写回 `user.json`。
12. 前端清空表单并提示修改成功。

### 7.5.1 设置或更新找回密码凭证流程

1. 已登录用户打开“设置找回密码凭证”表单。
2. 用户填写或确认 E-mail、找回问题和找回答案。
3. 前端校验 E-mail 格式、找回问题和找回答案不为空且长度合法。
4. 前端发送 `POST /api/auth/recovery-settings`。
5. 后端根据 Cookie 校验当前用户是否登录。
6. 后端读取 `user.json` 并找到当前用户。
7. 后端校验 E-mail 格式和唯一性。
8. 后端为找回答案生成新的随机盐和哈希。
9. 后端更新当前用户的 `email`、`recoveryQuestion`、`recoveryAnswerHash`、`recoveryAnswerSalt` 和 `recoveryUpdatedAt`。
10. 后端写回 `user.json`。
11. 前端清空表单并提示找回凭证已更新。

### 7.5.2 找回密码并重置密码流程

邮箱验证码主流程：

1. 未登录用户点击“忘记密码”。
2. 用户输入要重置的用户名，并输入任意接收验证码的 E-mail。
3. 前端发送 `POST /api/auth/password-reset-code`。
4. 后端读取 `user.json` 并找到用户。
5. 后端校验接收邮箱格式，并检查最近 60 秒内是否已发送验证码。
6. 后端生成 6 位验证码，为验证码生成随机盐和哈希。
7. 后端写入验证码哈希、盐、过期时间、发送时间、接收邮箱和错误次数 `0`。
8. 后端通过 SMTP 或邮件服务向用户本次输入的接收邮箱发送验证码。
9. 用户填写验证码、新密码和确认新密码。
10. 前端发送 `POST /api/auth/reset-password-by-code`。
11. 后端校验验证码是否存在、未过期、接收邮箱一致、错误次数未达到 3 次。
12. 验证码或接收邮箱错误时增加错误次数；第 3 次错误后清理验证码并返回错误。
13. 验证码正确时，为新密码生成新的随机盐和哈希。
14. 后端更新该用户的 `passwordHash` 和 `passwordSalt`，并清理验证码相关字段。
15. 前端提示密码已重置，请使用新密码登录。

找回问题兼容流程：

1. 未登录用户点击“忘记密码”。
2. 用户输入用户名。
3. 前端请求 `GET /api/auth/recovery-question?username=:username`。
4. 后端读取 `user.json`，找到用户并返回找回问题。
5. 用户填写找回答案、新密码和确认新密码。
6. 前端校验新密码格式和两次输入是否一致。
7. 前端发送 `POST /api/auth/reset-password`。
8. 后端读取 `user.json` 并找到用户。
9. 后端用该用户的 `recoveryAnswerSalt` 校验找回答案。
10. 校验失败时返回 `INVALID_RECOVERY_ANSWER`。
11. 校验成功后，为新密码生成新的随机盐和哈希。
12. 后端更新该用户的 `passwordHash` 和 `passwordSalt`，不改变用户 `id`、`username`、`role` 和投票记录。
13. 后端写回 `user.json`。
14. 前端提示密码已重置，请使用新密码登录。

### 7.5.3 邮件服务配置

邮箱验证码依赖 SMTP 或第三方邮件服务。MVP 建议使用 SMTP 配置：

- `SMTP_HOST`：SMTP 主机。
- `SMTP_PORT`：SMTP 端口。
- `SMTP_SECURE`：是否使用 TLS，值为 `true` 或 `false`。
- `SMTP_USER`：SMTP 用户名。
- `SMTP_PASS`：SMTP 密码或授权码。
- `SMTP_FROM`：发件人地址。

服务启动时可以从项目根目录 `.env` 文件读取以上配置；系统环境变量优先级高于 `.env` 文件。

项目应提供 SMTP 配置模板和检测脚本，降低真实邮箱接入成本：

- 提供 `smtp.env.example`，列出 Gmail、QQ 邮箱等常见配置写法和必填变量。
- 提供 `npm run check-smtp`，读取 `.env` 中的 SMTP 配置并发送一封测试邮件到 `SMTP_TEST_TO`。
- 检测脚本应在配置缺失、认证失败、端口或 TLS 配置错误时输出明确原因。
- 检测脚本不得打印 `SMTP_PASS` 明文。
- 真实邮箱授权码、应用专用密码或第三方 SMTP 密钥必须由项目维护者自行在邮箱服务商后台生成，不能由应用自动生成。
- 当真实 SMTP 配置通过 `npm run check-smtp` 验证后，业务接口 `POST /api/auth/password-reset-code` 应能返回 `codeSent: true`，用户应能在接收邮箱中收到验证码。

如果邮件配置缺失，发送验证码接口应返回 `EMAIL_SERVICE_NOT_CONFIGURED`，前端应提示管理员配置邮件服务。

### 7.6 管理员修改图书信息流程

1. 管理员登录。
2. 前端检查当前用户角色是否为 `admin`。
3. 管理员进入后台页面。
4. 前端请求 `GET /api/admin/books` 获取后台图书列表。
5. 管理员修改某本书的书名、作者、简介或封面地址并保存。
6. 前端发送 `PATCH /api/admin/books/:bookId`。
7. 后端校验用户是否登录且角色为 `admin`。
8. 后端校验图书是否存在，以及书名、作者、简介、封面地址是否合法。
9. 后端更新 `book.json` 中对应图书的 `title`、`author`、`description`、`coverUrl`。
10. 后端返回更新后的图书信息。
11. 前端刷新后台图书列表和前台展示数据。

### 7.7 管理员调整图书本数流程

新增图书流程：

1. 管理员在后台填写新图书名称和作者；简介和封面地址可选。
2. 前端发送 `POST /api/admin/books`。
3. 后端校验管理员权限、书名和作者。
4. 后端根据书名和作者联网抓取真实封面和简介。
5. 如果管理员手动提供简介或封面地址，手动内容优先。
6. 如果联网抓取失败且没有手动内容，后端使用本地匹配封面或默认封面、基础简介兜底。
7. 后端生成新图书 `id`，写入包含 `title`、`author`、`description`、`coverUrl` 的完整记录到 `book.json`。
8. 后端在 `vote.json.bookVotes` 中初始化该图书票数为 `0`。
9. 后端返回新增图书信息和元数据抓取结果。

删除图书流程：

1. 管理员在后台选择删除图书。
2. 前端展示二次确认。
3. 前端发送 `DELETE /api/admin/books/:bookId`。
4. 后端校验管理员权限和图书是否存在。
5. 后端从 `book.json` 中删除该图书。
6. 后端从 `vote.json.bookVotes` 中删除该图书票数。
7. 后端移除或清理 `vote.json.userVotes` 中该图书相关记录。
8. 后端返回删除成功结果。

### 7.8 管理员发起新一轮投票流程

1. 管理员登录并进入后台页面。
2. 管理员点击“发起新一轮投票”。
3. 前端展示二次确认，说明该操作会清空当前票数和所有用户投票记录。
4. 管理员确认后，前端发送 `POST /api/admin/vote-rounds`。
5. 后端校验当前用户是否登录且角色为 `admin`。
6. 后端读取 `book.json` 和 `vote.json`。
7. 后端将所有当前图书票数重置为 `0`。
8. 后端将 `vote.json.userVotes` 重置为空对象。
9. 后端写回 `vote.json`。
10. 前端刷新图书列表、后台图书列表和当前用户剩余票数。

## 8. 文件读写与数据一致性的基本处理思路

由于 MVP 使用本地 JSON 文件存储，需要注意文件读写的可靠性。

### 8.1 基础原则

- `book.json` 作为相对稳定的图书基础数据，主要用于读取。
- 管理员后台会修改 `book.json`，因此下一阶段需要对 `book.json` 写入做一致性处理。
- `vote.json` 作为频繁更新的数据文件，需要重点处理写入安全。
- 后端启动时可以检查 `book.json`、`vote.json`、`user.json` 是否存在，不存在时创建默认数据。
- 文件读取失败时，应返回明确错误，而不是让服务直接崩溃。

### 8.2 写入一致性

投票写入建议遵循以下流程：

1. 读取当前 `vote.json`。
2. 校验读取结果是否为合法 JSON。
3. 校验当前用户投票记录是否少于 3 条。
4. 修改指定图书票数。
5. 追加当前用户投票记录。
6. 将完整投票对象写回文件。
7. 写入成功后再向前端返回成功响应。

管理员修改图书信息建议遵循以下流程：

1. 读取当前 `book.json`。
2. 校验当前用户是否为管理员。
3. 校验图书数据和请求参数。
4. 将完整图书数组写回 `book.json`。
5. 如涉及新增或删除图书，同步更新 `vote.json`。
6. 所有文件写入成功后再向前端返回成功响应。

### 8.3 并发处理

MVP 阶段可以采用简单队列或内存锁控制投票写入，避免多个投票请求同时写入导致数据覆盖。

建议思路：

- 同一时间只允许一个写入操作修改 `vote.json`。
- 管理员修改 `book.json` 时，也应使用队列或内存锁，避免和读取、删除、投票写入产生数据竞争。
- 后续投票请求排队等待前一个写入完成。
- 写入失败时不更新前端票数，并返回错误。

### 8.4 数据备份

虽然 MVP 可以不实现复杂备份，但建议在后续迭代中考虑：

- 写入前保留上一版 `book.json`、`vote.json`、`user.json`。
- 写入失败时恢复备份。
- 定期导出投票结果。

### 8.5 用户数据与会话处理

- `user.json` 需要和 `vote.json` 一样考虑写入一致性，避免并发注册时覆盖用户数据。
- 注册写入建议使用简单队列或内存锁，保证同一时间只有一个写入操作修改 `user.json`。
- 修改密码写入也应复用同一队列或内存锁，避免和注册写入互相覆盖。
- 设置找回凭证和重置密码写入也应复用同一队列或内存锁，避免和注册、修改密码写入互相覆盖。
- Session 可以保存在服务端内存中，key 为随机 `sessionId`，value 为用户基础信息或用户 ID。
- Session Cookie 应设置 `HttpOnly`，建议设置 `SameSite=Lax`。
- MVP 阶段可以不实现 Session 持久化，服务重启后用户需要重新登录。

### 8.6 图书元数据联网抓取处理

- 图书元数据抓取应封装为独立模块，避免后台新增图书逻辑直接依赖具体数据源。
- 图书元数据抓取输入为 `title` 和 `author`，输出为可选的 `description`、`coverUrl`、`source`、`confidence`。
- 新增图书和刷新已有封面时应优先联网查询公开图书信息源，目标是获取真实封面和真实图书简介。
- 数据源可以配置多个候选源，建议优先级为：可访问的搜索结果结构化数据、公开图书 API、本地内置元数据、默认兜底。
- 当前实现应包含可从本机访问的结构化搜索源，避免只依赖 Google Books 或 Open Library 等可能连接超时的数据源。
- 抓取请求必须设置超时时间，避免后台新增图书长时间阻塞；单个数据源超时后应尝试下一个数据源。
- 后端应记录抓取来源和匹配状态，并通过响应 `metadata` 返回 `source`、`confidence`、`warnings`。
- 抓取到的远程封面 URL 如不稳定，后端可以下载并缓存到本地静态目录，再将本地 URL 写入 `book.json`。
- 刷新已有封面时只更新 `coverUrl`，不应自动覆盖已有简介，避免把作者介绍或版本说明误写入简介。
- 联网抓取失败不应导致新增图书整体失败，除非兜底默认封面和基础简介也无法生成。
- 后端写入 `book.json` 前必须保证新图书包含非空 `title`、`author`、`description`、`coverUrl`。
- 前端展示封面时应使用适合竖版封面的样式，避免封面被裁剪成不可识别的横条。

## 9. 后续可扩展方向

下一阶段需求完成后，可以按以下方向扩展：

- 更细粒度投票限制：限制每个用户、IP 或设备每天投票次数。
- 排行榜：按票数从高到低展示图书排名。
- 管理后台增强：支持管理员维护分类、标签、排序等更多图书字段。
- 数据库迁移：将 JSON 文件存储升级为 SQLite、MySQL、PostgreSQL 或 MongoDB。
- 投票记录：记录每次投票时间、用户标识和来源信息。
- 图表统计：展示票数占比、趋势图等统计信息。
- 搜索和筛选：支持按书名、作者、分类筛选图书。
- 移动端适配：优化手机浏览体验。
- 部署支持：增加生产环境启动、日志和错误监控方案。
